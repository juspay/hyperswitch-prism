# UCS Error Handling: Current State & Unification Options

**Purpose:** Document current error handling inconsistency and solution options
**Date:** 2026-03-30

---

## The Inconsistency (TL;DR)

UCS has two error types:

- **Connector Business Errors** (card declined, insufficient funds, invalid CVV) — generated by connector integrations as `ErrorResponse`, should populate `ErrorInfo` in the response proto.
- **UCS Integration Errors** (missing required fields, bad config, proto parsing) — infrastructure/integration failures on the UCS side.

Both types are correctly handled for connector business errors across all flows. The inconsistency is in how **integration errors** are handled:

| Flow | Integration Errors | Wire Format |
|------|-------------------|-------------|
| **Authorize** | Converted to `ErrorResponse` → populates `ErrorInfo` | `Ok(Response)` — always HTTP 200 |
| **Capture / Refund / Void / PSync / etc.** | Propagated as `Err(tonic::Status)` | `Err(Status)` — HTTP 400/500 |

**Root cause:** Authorize was hand-written in `payments.rs` using domain-first error modeling (`PaymentAuthorizationError`). All other flows are macro-generated in `utils.rs` and follow the standard gRPC `Err(Status)` pattern. Both are valid individually; the problem is they co-exist.

## How Each Flow Works Today

### Authorize (`payments.rs`)

```rust
async fn process_authorization_internal(...) -> Result<PaymentServiceAuthorizeResponse, PaymentAuthorizationError> {
    let response = execute_connector_processing_step(...).await;

    match response {
        Ok(success_response) => {
            // Business errors already in router_data.response = Err(ErrorResponse)
            generate_payment_authorize_response(success_response)?
        }
        Err(error_report) => {
            // Integration errors → manually construct ErrorResponse → populate ErrorInfo
            let app_err: ApplicationErrorResponse = error_report.current_context().switch();
            let error_router_data = RouterDataV2 {
                response: Err(ErrorResponse { code: app_err.sub_code, message: app_err.error_message, ... })
            };
            generate_payment_authorize_response(error_router_data)?
        }
    }
}

pub async fn authorize(...) -> Result<Response<PaymentServiceAuthorizeResponse>, Status> {
    match process_authorization_internal(...).await {
        Ok(response) => Ok(Response::new(response)),   // Always Ok — errors in ErrorInfo
        Err(e) => Ok(Response::new(PaymentServiceAuthorizeResponse::from(e))), // Also Ok
    }
}
```

All errors (business and integration) end up in `ErrorInfo`. Returns `Ok(tonic::Response)` for every case.

### Other Flows — Capture, Refund, Void, PSync (`utils.rs` macro)

```rust
// Generated by implement_connector_operation! macro
pub async fn capture(...) -> Result<Response<...>, Status> {
    let result = execute_connector_processing_step(...)
        .await
        .switch()
        .into_grpc_status()?;   // ← Integration errors become Err(tonic::Status)

    let response = generate_capture_response(result).into_grpc_status()?;
    Ok(Response::new(response))
}
```

Integration failures propagate as `Err(tonic::Status)` with an appropriate gRPC code (e.g. `InvalidArgument`).

## Downstream Impact on HTTP Clients & Hyperswitch

### HTTP Clients (Axum layer)

The Axum HTTP handlers (`http/handlers/macros.rs`) call the gRPC service directly. `tonic::Status` is mapped to HTTP codes in `http/error.rs`:

| `tonic::Code` | HTTP Status |
|--------------|-------------|
| `InvalidArgument` | 400 Bad Request |
| `NotFound` | 404 Not Found |
| `Internal` | 500 Internal Server Error |
| `Unavailable` | 503 Service Unavailable |

Result:
- **Authorize integration errors** → HTTP 200 with `{"error": {...}}` in body
- **Capture/Refund integration errors** → HTTP 400 or 500

### Hyperswitch (for the HS↔UCS integration)

| Path | What HS receives | Final API error |
|------|-----------------|-----------------|
| **UCS Authorize** (integration error) | `Ok(proto)` with error fields → `RouterData.response = Err(ErrorResponse{code, message})` | `ExternalConnectorError { code: "MISSING_FIELD", message: "..." }` — details preserved as strings |
| **UCS Capture/etc** (integration error) | `Err(tonic::Status)` → `ConnectorError::ResponseHandlingFailed` | `InternalServerError` — details lost |
| **UCS any flow** (connector business error) | `Ok(proto)` with error fields → `RouterData.response = Err(ErrorResponse)` | `ExternalConnectorError` — same as authorize above |

All three paths have different error granularity compared to the direct connector path (which produces structured `ApiErrorResponse` variants like `MissingRequiredField { field_name }`). This is a known structural difference between direct and UCS paths, not something to fix in UCS.

```
      │
      ▼

      │
      ▼

```

## Solution Options

### Option 1: The Strict Two-Layer Model (Standard API Design) (RECOMMENDED)

This approach respects standard API protocol boundaries by separating "Integration/API failures" from "Business/Connector failures".

**a. Integration/Contract Errors (UCS-side):** Bad JSON, MissingRequiredField, AmountConversionFailed.

**gRPC Action:** Return Err(tonic::Status::invalid_argument).

**HTTP Behavior:** The Axum layer automatically translates this to an HTTP 400 Bad Request (or 500 Internal Server Error).

**Why:** The request was malformed or the integration failed; the request never successfully made it through the business logic.

**b. Business/Connector Errors (Connector-side):** Card declined, invalid CVV, connector explicitly rejects the payload.

**gRPC Action:** Return Ok(Response) and populate the ErrorInfo field (e.g., connector_details.code = "DECLINED").

**HTTP Behavior:** HTTP 200 OK, with the error payload gracefully handled in the JSON body.

**Why:** The gRPC call succeeded and the server executed perfectly, but the payment itself failed at the downstream connector level.

**Changes:**

- Enhance Hyperswitch `UnifiedConnectorServiceError` handling
- Map `tonic::Code` to specific `ConnectorError` variants:
  - `InvalidArgument` → `MissingRequiredField`
  - `FailedPrecondition` → `FlowNotSupported`
  - `Unavailable` → Retryable error class
- Update `to_payment_failed_response` to preserve these mappings

**Cons**
- Loses rich error structure (limited to tonic::Code + message string) — But, this is actually solvable. Tonic supports the gRPC "Rich Error" model. We can serialize our ErrorInfo protobuf into a byte buffer and attach it directly to the tonic::Status using Status::with_details(). However, this requires extra work on the client side to deserialize the grpc-status-details-bin header.
- Harder to add payment-specific metadata

(Alternatively, in the long term, we could refactor the core logic to return a unified ApiErrorResponse, from which both HTTP and gRPC layers independently derive their respective formatting)

Note: Business error from external connector that is parsed from our connector transformer will still to ErrorInfo of Ok grpc response.

### Option 2: The Domain-First Model

This approach puts everthing (business errors from connector as earlier, IntegrationError, ConnectorResponseTransformError ) inside the ErrorInfo protobuf message.
(as done in Current Authorize)

**Integration/Contract Errors (UCS-side):** Client forgot the "amount" field.

**gRPC Action:** Return Ok(Response) and populate ErrorInfo (e.g., unified_details.code = "MISSING_REQUIRED_FIELD", error_source = UCS).

**Business/Connector Errors (Connector-side):** Card declined.

**gRPC Action:** Return Ok(Response) and populate ErrorInfo (e.g., connector_details.code = "DECLINED", error_source = CONNECTOR).

**The HTTP Catch:** Because everything returns Ok(), the HTTP layer will default to returning 200 OK for everything. To maintain REST standards, the Axum handlers must explicitly inspect the error_source and manually override the HTTP status to 400 Bad Request if the source is UCS.

(Alternatively, in the long term, we could refactor the core logic to return a unified ApiErrorResponse, from which both HTTP and gRPC layers independently derive their respective formatting)

**Metrics Skew:** Standard gRPC monitoring (Prometheus/Datadog) will log bad configurations and missing fields as "Successful" RPC calls, requiring us to build custom metrics tracking based on the payload.
*(Alternatively, in the long term, we could refactor the core logic to return a unified `ApiErrorResponse`, from which both HTTP and gRPC layers independently derive their respective formatting).*

> ### ℹ️ Context for Option 2: Why Differentiate UCS vs Connector Errors?
> *(Note: This distinction is strictly required for Option 2. Option 1 handles this natively via transport status codes).*
>
> An important consideration for any unified error handling approach is distinguishing between **two fundamentally different error sources:**
>
> **Error Source Types**
>
> | Error Source | Cause | Example | Actionable By |
> |--------------|-------|---------|---------------|
> | **UCS-side** | Integration or infrastructure issues | HS didn't send required field, bad connector config, proto parsing failure | HS or UCS team |
> | **Connector-side** | Normal business outcomes from external connector | Card declined, insufficient funds, invalid CVV | Merchant (retry/different card) |
>
> **Why This Matters**
>
> 1. **Monitoring & Alerting:** **UCS errors** should trigger alerts (integration bugs, need fixing). **Connector errors** are normal business behavior (don't alert, just track rate).
> 2. **SLO Definition:** UCS availability SLO should **exclude** connector errors. A connector declining cards is not a UCS failure.
> 3. **Debugging & Analytics:** Quickly identify if an issue is an HS<>UCS integration bug vs normal payment failure, and track connector-specific error rates for analytics.
>
> **Implementation Approach (Required for Option 2)**
> Add an `error_source` field to error details:
>
> ```protobuf
> message UnifiedErrorDetails {
>   optional string code = 1;
>   optional string message = 2;
>   optional string description = 3;
>   optional string user_guidance_message = 4;
>   optional ErrorSource source = 5;  // NEW field
> }
>
> enum ErrorSource {
>   ERROR_SOURCE_UNSPECIFIED = 0;
>   ERROR_SOURCE_CONNECTOR = 1;  // Error from external connector API
>   ERROR_SOURCE_UCS = 2;         // Error from UCS infra/integration
> }
> ```
