## PART 1: DYNAMIC PREREQUISITE DETECTION & WIRING

### 1.1 How to Detect Prerequisites

When testing a flow, analyze field_probe data to detect dependencies:

**Method 1: Error Message Analysis**
The `error` field in field_probe entries tells you exactly what's missing:
```
"authorize/Card": {
  "status": "error",
  "error": "Stuck on field: payment_method_token — Missing required field: payment_method_token (from PaymentMethodToken flow)"
}
```
This tells you: authorize needs `payment_method_token` → run tokenize first.

**Method 2: Proto Request Field Analysis**
Check the `proto_request` for required fields that come from other flows:
```python
target_proto = field_probe['flows']['token_authorize']['default']['proto_request']
if 'connector_token' in target_proto:
    prereq.append('tokenize')
if 'customer' in target_proto and 'id' in target_proto['customer']:
    prereq.append('create_customer')
```

**Method 3: Supported Flow Pattern Matching**
```
IF authorize/{PaymentMethod} has status "error"
AND token_authorize/default has status "supported"
AND tokenize/default has status "supported"
THEN this connector requires tokenization flow
```

### 1.2 Prerequisite Dependency Graph

| TARGET FLOW         | PREREQUISITES (in order)                       |
|---------------------|------------------------------------------------|
| authorize           | (none for direct connectors) / create_customer → tokenize (for token-based) |
| token_authorize     | create_customer → tokenize                     |
| capture             | authorize (needs connector_transaction_id)     |
| refund              | authorize → capture (needs txn_id)             |
| void                | authorize (needs connector_transaction_id)     |
| get (psync)         | authorize (needs connector_transaction_id)     |
| refund_get (rsync)  | refund (needs connector_refund_id)             |
| incremental_auth    | authorize (manual capture, needs txn_id)       |

### 1.3 Automatic Data Wiring Between Flows

```bash
# Step 1: Create Customer
CUST_RESPONSE=$(grpcurl ... types.CustomerService/Create)
CUSTOMER_ID=$(echo "$CUST_RESPONSE" | jq -r '.connector_customer_id // .merchantCustomerId')

# Step 2: Tokenize (inject customer_id)
TOK_REQUEST=$(cat data/field_probe/{connector}.json | jq '.flows.tokenize.default.proto_request')
TOK_REQUEST=$(echo "$TOK_REQUEST" | jq --arg cid "$CUSTOMER_ID" '.customer.id = $cid')
TOK_RESPONSE=$(grpcurl ... -d "$TOK_REQUEST" types.PaymentMethodService/Tokenize)
TOKEN=$(echo "$TOK_RESPONSE" | jq -r '.payment_method_token // .paymentMethodToken')

# Step 3: Authorize (inject token)
AUTH_REQUEST=$(cat data/field_probe/{connector}.json | jq '.flows.token_authorize.default.proto_request')
AUTH_REQUEST=$(echo "$AUTH_REQUEST" | jq --arg tok "$TOKEN" '.connector_token = $tok')
AUTH_RESPONSE=$(grpcurl ... -d "$AUTH_REQUEST" types.PaymentService/Authorize)
TXN_ID=$(echo "$AUTH_RESPONSE" | jq -r '.connector_transaction_id // .connectorTransactionId')

# Step 4: Capture (inject transaction_id)
CAP_REQUEST='{"connector_transaction_id": "'$TXN_ID'", "minor_amount": 1000, "currency": "USD"}'
CAP_RESPONSE=$(grpcurl ... -d "$CAP_REQUEST" types.PaymentService/Capture)
```

### 1.4 Field Mapping Reference

| Source Flow     | Output Field             | Target Flow                        | Input Field              |
|-----------------|--------------------------|------------------------------------|--------------------------|
| create_customer | connector_customer_id    | tokenize                           | customer.id              |
| tokenize        | payment_method_token     | authorize                          | connector_token          |
| authorize       | connector_transaction_id | capture, void, get, incremental_auth | connector_transaction_id |
| authorize       | connector_transaction_id | refund                             | connector_transaction_id |
| refund          | connector_refund_id      | refund_get                         | connector_refund_id      |

## PART 2: INTELLIGENT STATUS HANDLING

### 2.1 Status is a HINT, Not a Blocker

| Status          | Meaning                                      | Action                                                      |
|-----------------|----------------------------------------------|-------------------------------------------------------------|
| supported       | Field probe generated valid request          | Test with high confidence                                   |
| not_implemented | Flow not built in Prism yet                  | Attempt if explicitly requested, expect failure             |
| not_supported   | Field probe thinks connector doesn't support | ATTEMPT ANYWAY — may be wrong for new flows                 |
| error           | Field probe couldn't generate request        | Check error message — may indicate prerequisites needed     |

### 2.2 When to Override Status

ALWAYS attempt testing when:
1. User explicitly requests a specific flow
2. The flow is new and field_probe may be stale
3. Error message indicates prerequisites (not a real failure)

### 2.3 Interpreting `error` Status Correctly

The `error` status often means "needs prerequisites", not "broken". If the error mentions a missing field that is the output of another flow, run that flow first.

## PART 3: RESULT REPORTING WITH MASKING

### 3.1 What to Report

For EVERY test (pass or fail), report:
1. grpcurl command (exact, creds masked)
2. Raw connector request (from `rawConnectorRequest`, masked)
3. Raw connector response (from `rawConnectorResponse`, masked)
4. Status and verdict

### 3.2 Sensitive Data Masking Rules

MUST MASK:
- API Keys/Secrets: `x-api-key` → `<REDACTED_API_KEY>`; `Authorization: Bearer|Basic xxx` → `<REDACTED>`; `api_key` in body → `<REDACTED_API_KEY>`
- Card Data: full PAN → `<REDACTED_PAN>` (or show last 4: `****1111`); CVV → `<REDACTED_CVV>`; cryptogram → `<REDACTED_CRYPTOGRAM>`
- Tokens: `payment_method_token`, `access_token`, `connector_token` → `<REDACTED_TOKEN>`
- Personal: SSN/Tax ID → `<REDACTED>`; bank account numbers → `<REDACTED_ACCOUNT>`

KEEP VISIBLE: test emails, test names, transaction IDs, status codes, error messages, amounts, currencies, timestamps, routing numbers.

### 3.3 Masking Implementation

```bash
mask_sensitive() {
  local input="$1"
  echo "$input" | \
    sed -E "s/(x-api-key: )[^'\"]+/\1<REDACTED_API_KEY>/gi" | \
    sed -E "s/(Authorization: Bearer )[^'\"]+/\1<REDACTED>/gi" | \
    sed -E "s/(Authorization: Basic )[^'\"]+/\1<REDACTED>/gi" | \
    sed -E "s/[0-9]{13,19}/<REDACTED_PAN>/g" | \
    sed -E "s/(card_cvc|cvc|cvv)[\"': ]+[0-9]{3,4}/\1\": \"<REDACTED_CVV>\"/gi" | \
    sed -E "s/(payment_method_token|connector_token|access_token)[\"': ]+[^\"']+/\1\": \"<REDACTED_TOKEN>\"/gi" | \
    sed -E "s/(cryptogram|online_payment_cryptogram)[\"': ]+[^\"']+/\1\": \"<REDACTED_CRYPTOGRAM>\"/gi"
}
```

## PART 4: COMPLETE WORKFLOW

### 4.1 When Given a Flow to Test

1. Load `data/field_probe/{connector}.json`.
2. Check flow status and detect prerequisites (Parts 1.1–1.2).
3. Execute prerequisite chain with data wiring (Part 1.3).

### 4.2 Error Classification

| Symptom                              | Root Cause            | Report As                   |
|--------------------------------------|-----------------------|-----------------------------|
| grpcurl: method not found            | Wrong service name    | AGENT_CONFIG_ERROR          |
| grpcurl: malformed JSON              | Bad request build     | AGENT_CONFIG_ERROR          |
| grpcurl: connection refused          | Server not running    | ENVIRONMENT_ERROR           |
| Prism: 401/403                       | Bad credentials       | CREDENTIAL_ERROR            |
| Prism: 400 + "missing field X"       | Incomplete proto      | PREREQUISITE_NEEDED         |
| Connector: 401/403                   | Connector auth fail   | CONNECTOR_CREDENTIAL_ERROR  |
| Connector: 400 + business error      | Connector rejection   | CONNECTOR_REJECTED          |
| Connector: 200 + CHARGED             | Full success          | PASS                        |
| Connector: 200 + AUTHORIZED          | Manual capture OK     | PASS                        |
| Connector: 200 + PENDING             | Async processing      | PASS_PENDING                |
| Connector: 200 + redirect URL        | Redirect flow         | PASS_REDIRECT               |
| Connector: 200 + FAILED              | Business rejection    | CONNECTOR_REJECTED          |
| Field probe: not_supported           | May be stale          | ATTEMPT_ANYWAY              |

## PART 8: WEB SEARCH FALLBACK FOR TEST PAYMENT DATA

### 8.1 When to Use Web Search

Use web search ONLY when:
1. Standard test data from field_probe fails with connector-specific rejection
2. Error indicates invalid card/account number (not missing fields or auth errors)
3. You've already tried the default test data at least once

Error patterns that trigger web search:
- "Invalid card number"
- "Card not accepted"
- "Test card not recognized"
- "Invalid account number"
- "Invalid routing number"
- "Card BIN not supported"
- "Payment method not valid for sandbox"
- "Use sandbox test credentials"

### 8.2 What CAN Be Web-Searched

ALLOWED — Payment method instrument data only:

Card payments:
- `card_number` (test card numbers)
- `card_exp_month` / `card_exp_year` (test expiry)
- `card_cvc` (test CVV)

Bank debits:
- `account_number` (test account)
- `routing_number` (test routing — ACH)
- `sort_code` (test sort code — BACS)
- `bsb_number` (test BSB — BECS)
- `iban` (test IBAN — SEPA)
- `bank_code` (test bank codes)

Wallets (decrypted test data):
- `application_primary_account_number` (test DPAN)
- `online_payment_cryptogram` (test cryptogram)

### 8.3 What MUST NEVER Be Web-Searched

FORBIDDEN — These come from gRPC flows or are auto-generated:

From gRPC responses (wired automatically):
- `connector_transaction_id`
- `connector_refund_id`
- `connector_customer_id`
- `payment_method_token` / `connector_token`
- `order_id` (from CreateOrder flow)
- `authorization_id`
- `capture_id`

Auto-generated:
- `merchant_transaction_id` (you generate this)
- `request_id`
- `idempotency_id`
- `reference_id`

### 8.4 Web Search Workflow

1. **Try with field_probe default test data** — run grpcurl with `proto_request` as-is. If SUCCESS, done. If FAIL with "invalid card" type error, continue.
2. **Identify what needs to be searched** — parse the error message: card? account? routing? Note the connector name.
3. **Perform targeted web search** (in priority order):
   1. `"{connector} sandbox test card numbers site:docs.{connector}.com"`
   2. `"{connector} api test credit card"`
   3. `"{connector} developer test payment data"`
   4. `"{connector} sandbox test account numbers"` (for bank debits)
4. **Extract test data from search results** — official documentation first; extract specific test card/account numbers; note any special requirements (specific CVV, expiry, etc.).
5. **Retry with web-searched test data** — replace ONLY the payment method instrument fields; keep all other fields from field_probe `proto_request`; run grpcurl again.
6. **Report the source** — if web-searched data was used: `TEST DATA SOURCE: Web search (official {connector} docs)`.

### 8.8 Rules for Web Search

1. **Only search for payment instrument data** — never for IDs, tokens, or references.
2. **Prefer official documentation** — `docs.{connector}.com` or `developer.{connector}.com`.
3. **One retry only** — if web-searched data also fails, report as `CONNECTOR_REJECTED`.
4. **Document the source** — always note where test data came from.
5. **Cache successful searches** — remember working test data for future runs on the same connector.
6. **Don't search for connector credentials** — those come from `creds.json` only.
7. **Validate search results** — card numbers 13–19 digits, routing numbers 9 digits, etc.

### 8.9 Integration with Main Workflow

Updated main test flow:
1. Load field_probe `proto_request`.
2. Execute prerequisites (create_customer, tokenize, etc.).
3. Attempt target flow with default test data.
4. IF PASS → report success.
5. IF FAIL with "invalid card/account" error:
   a. Perform web search for connector-specific test data.
   b. Replace ONLY payment method instrument fields.
   c. Retry the flow (one time only).
   d. Report result with test data source noted.
6. IF FAIL with other errors → report failure with full details.

### 8.10 Summary

| Aspect             | Rule                                                                       |
|--------------------|----------------------------------------------------------------------------|
| When to search     | Only after default test data fails with "invalid card/account" type errors |
| What to search     | Card numbers, account numbers, routing numbers, IBANs, BSBs, etc.          |
| What NOT to search | Transaction IDs, tokens, customer IDs, order IDs — these come from gRPC    |
| Search priority    | Official docs first, then developer forums, then general results           |
| Retry limit        | One retry with web-searched data, then report failure                      |
| Documentation      | Always note `TEST DATA SOURCE: Web search ({source})` in report            |
