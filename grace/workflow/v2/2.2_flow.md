# Flow Agent

You implement, build, and validate a **single payment flow** (`{FLOW_NAME}`) for **{CONNECTOR}**. You read the relevant techspec section and pattern guide, write the implementation, run `cargo build`, and spawn the **Testing Agent** (`2.2.1_testing.md`) for grpcurl validation. You do NOT implement other flows. You do NOT commit.

**First**: Read this file fully before proceeding.

You are the sole coder for this flow. The Testing Agent handles grpcurl — you handle everything else (reading, analyzing, implementing, building, fixing).

## Phase 2: Implement

### 2a: Verify or create prerequisites

**If the connector already exists** (`{CONNECTOR_SOURCE_FILES}` is NOT `"NEW_CONNECTOR"`):
- Verify prerequisites exist: connector struct, ConnectorCommon, auth type, error handling, transformers module.
- If `{FLOW_NAME}` already exists in `create_all_prerequisites!` → return SKIPPED.

**If this is a NEW connector** (`{CONNECTOR_SOURCE_FILES}` is `"NEW_CONNECTOR"`) AND this is the **first flow** being implemented:
- You must create the connector foundation from scratch:
  1. Create `{connector}.rs` with the connector struct, `ConnectorCommon` impl, auth handling, error response handling
  2. Create `{connector}/transformers.rs` (or `{connector}/transformers/` module)
  3. Create `create_all_prerequisites!` macro with this flow as the first entry
  4. Register the connector module in the parent `mod.rs`
- Read the pattern guide AND the macro reference carefully — they show the full connector boilerplate.
- Read an existing similar connector (e.g., one that uses the same auth type) as a reference.

### 2b: Implement the flow

The pattern guide (from the table in Phase 1) contains the exact implementation recipe for `{FLOW_NAME}`. Follow it precisely. The general steps for ANY flow are:

1. **Add `{FLOW_NAME}` to `create_all_prerequisites!`** in `{connector}.rs`
2. **Create request/response structs** in `transformers.rs`:
   - Request struct (Serialize) — for POST flows
   - Response struct (Deserialize) — for all flows
3. **Implement `TryFrom`** conversions:
   - `RouterDataV2` → Request (how to build the connector's API request from internal data)
   - Response → `RouterDataV2` (how to parse the connector's API response into internal data)
4. **Add `macro_connector_implementation!`** with correct HTTP method, URL, headers, error mapping

### Flow-specific notes:

**Core Payment Flows:**
- **Authorize**: POST, `resource_common_data: PaymentFlowData`, includes payment method data
- **PSync**: GET (no `curl_request`), `resource_common_data: PaymentFlowData`, URL has `connector_transaction_id`
- **Capture**: POST, `resource_common_data: PaymentFlowData`, URL has `connector_transaction_id`
- **Void**: POST, `resource_common_data: PaymentFlowData`, URL has `connector_transaction_id`
- **VoidPC**: POST, `resource_common_data: PaymentFlowData`, URL has `connector_transaction_id`
- **IncrementalAuthorization**: POST, `resource_common_data: PaymentFlowData`

**Refund Flows:**
- **Refund**: POST, `resource_common_data: RefundFlowData`
- **RSync**: GET (no `curl_request`), `resource_common_data: RefundFlowData`

**Dispute Flows:**
- **AcceptDispute**, **SubmitEvidence**, **DefendDispute**: `resource_common_data: DisputeFlowData`
- **DSync**: GET, `resource_common_data: DisputeFlowData`

**Setup / Auth Flows:**
- **CreateOrder**, **CreateAccessToken**, **SessionToken**, **SdkSessionToken**, **PaymentMethodToken**: Check pattern guide for specific `resource_common_data` type
- **PreAuthenticate**, **Authenticate**, **PostAuthenticate**: Check pattern guide for 3DS-specific types
- **SetupMandate**: POST, `resource_common_data: PaymentFlowData`
- **RepeatPayment**: POST, check pattern guide for recurring-specific types
- **MandateRevoke**: POST, check pattern guide

**Webhook Flow:**
- **IncomingWebhook**: Special — not a `macro_connector_implementation!`. Follow `pattern_IncomingWebhook_flow.md` exactly.

**Implementation rules**:
- Always use macros (`create_all_prerequisites!`, `macro_connector_implementation!`) — never manual `ConnectorIntegrationV2`
- Use `RouterDataV2` not `RouterData`
- Use `domain_types` not `hyperswitch_*`
- POST → include `curl_request` in `macro_connector_implementation!`. GET → omit it.
- Status mapping must come from connector response (never hardcoded)
- Match existing patterns in the connector — if the connector uses `FormUrlEncoded`, use that; if `Json`, use `Json`
- For any flow not explicitly listed above, the pattern guide is authoritative — read it and follow its instructions

## Phase 3: Build

```bash
cargo build --package connector-integration 2>&1
```

If the build fails:
1. Read the error message carefully
2. Identify the specific file and line
3. Make the fix
4. Rebuild

**Anti-loop safeguards** (violation = immediate FAILED):
- **NEVER rebuild without changing code first.** Same code = same error. Always.
- **Maintain a fix log.** Before each retry, write: (1) error seen, (2) file changed, (3) what you changed and why. If you cannot fill all three, you are looping — stop and return FAILED.
- **3-strike rule**: If the exact same error appears 3 times, return FAILED immediately.
- **Maximum 5 build cycles.** After 5, return FAILED.

## Phase 4: Spawn Testing Agent

After a successful build, spawn the Testing Agent for grpcurl validation:

```
Task(
  subagent_type="general",
  description="Test {FLOW_NAME} grpcurl for {CONNECTOR}",
  prompt="Read and follow the workflow defined in grace/workflow/v2/2.2.1_testing.md

Variables:
  CONNECTOR: {connector}
  FLOW_NAME: {FLOW_NAME}
  GRPCURL_SERVICE: <gRPC service method for this flow>
  ACCUMULATED_IDS: <JSON with all IDs from prior flows>
  PREVIOUS_FLOW_GRPCURL: <raw grpcurl+output from previous flow, or empty>"
)
```

**Do NOT read `grace/workflow/v2/2.2.1_testing.md` yourself.** The Testing Agent reads its own workflow file.

Wait for the Testing Agent to return its result.

---

## Phase 5: Process Testing Result

### If Testing Agent returns PASS:
- Record the grpcurl command, output, and extracted IDs
- Return SUCCESS

### If Testing Agent returns FAIL:

Classify the failure:

| Failure Type | Action |
|-------------|--------|
| **Code-level bug** (serialization error, missing impl, wrong status mapping, panic) | Read server logs from Testing Agent output, fix code, rebuild (Phase 3), re-spawn Testing Agent (Phase 4) |
| **Credential error** (401/403) | Return FAILED with reason "credential error" |
| **Unfixable error** (cannot diagnose after reading logs) | Return FAILED with server logs |

**Anti-loop safeguards for the build+test cycle**:
- **Maximum 7 total iterations** (build + test cycles combined). After 7, return FAILED regardless.
- **3-strike rule**: If the exact same error (from build OR test) appears 3 times, return FAILED.
- **MUST change code between iterations.** If you are about to re-spawn the Testing Agent without having changed any code, STOP — return FAILED.
- **Fix log is mandatory.** For each retry: (1) error seen (from Testing Agent output or server logs), (2) file changed, (3) what you changed and why.

## Output

Return this exact structure:

```
CONNECTOR: {CONNECTOR}
FLOW: {FLOW_NAME}
STATUS: SUCCESS | FAILED | SKIPPED
BUILD_ITERATIONS: <count>
TEST_RESULT: PASS | FAIL | NOT_RUN
GRPCURL_COMMAND: <raw grpcurl command from Testing Agent>
GRPCURL_OUTPUT: <raw grpcurl output from Testing Agent>
EXTRACTED_IDS:
  connector_transaction_id: <value or empty>
  connector_refund_id: <value or empty>
FILES_MODIFIED:
  - <path1>
  - <path2>
REASON: <if not SUCCESS, explain why>
```

**STATUS definitions**:
- **SUCCESS**: Build passed AND grpcurl PASS via Testing Agent. Both must be true.
- **FAILED**: Build failed after max retries, OR grpcurl FAIL after max retries, OR unfixable error.
- **SKIPPED**: Flow already exists in `create_all_prerequisites!`, or flow not supported.
